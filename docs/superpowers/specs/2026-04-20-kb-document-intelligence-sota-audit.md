# KB / Document Intelligence — SOTA Audit + Spec

## TL;DR

Using OpenRouter-reachable models only (no GPU available locally), we benchmarked the current RantAI-Agents knowledge-base pipeline against every reachable SOTA alternative. The current pipeline has **three high-confidence wins** available, ordered by impact-per-engineering-effort:

1. **Replace `unpdf` with a vision-LLM extractor for PDFs.** `unpdf` produces 0 headings, 0 tables, 0 equations on a 15-page arxiv paper. Every vision-LLM tested produces proper markdown structure. Cascades into chunking quality — without paragraph/heading breaks, `SmartChunker` silently falls back to sentence splitting, throwing away section metadata.

   **Cost-aware pick (default):** `openai/gpt-4.1-nano` at **$0.00031/page** (~$87/month for 5K pages/day). Measured 2-4s latency, clean markdown with proper headings, preserves all numerical data on the financial-table stress doc. Fallback: `google/gemini-3-flash-preview` at $0.0024/page — best table structure fidelity, kicks in when the primary fails.

   **Avoided:** `gemini-2.5-flash` and `gemini-2.5-flash-lite` without the compact-tables prompt fix (they emitted 400K+ char padded tables for a 4-page doc). The repo's current `VisionLlmExtractor` prompt includes the compact fix, so either is safe to pick via `KB_EXTRACT_PRIMARY`. `amazon/nova-lite-v1`, `openai/gpt-5-nano`, `bytedance-seed/seed-1.6-flash`, `qwen3-vl-{8b,32b,235b}`, `pixtral-large` either returned 0 chars, timed out, or were more expensive than `gpt-4.1-nano` for comparable quality.

2. **Swap the embedding model** from `openai/text-embedding-3-small` → **`qwen/qwen3-embedding-8b`**. Measured: +14.3 pts hit@1 (0.771 → 0.914), +7.9 pts MRR (0.864 → 0.943), 4096-dim, and actually *faster* for batch embedding. Cost is ~2× (still $0.04/M tokens — negligible). Biggest single win on retrieval accuracy.

3. **Add LLM-as-reranker using `google/gemini-3-flash-preview` after top-20 cosine — NOT Cohere rerank.** Measured: +5.7 pts hit@1 (0.914 → 0.971), +2.9 pts MRR, **+1700ms latency**. **Counterintuitive finding**: `cohere/rerank-v3.5` actually *hurt* hit@1 (-5.7 pts) when the base embedder is already strong — it demotes precise-answer chunks in favor of "introduction to the answer" chunks. LLM-rerank with the question in context doesn't make that mistake.

Secondary wins (lower priority):
4. Normalize BOTH the embedding index and the query side to carry **section headers + document title as prefix** when embedding (code already does this for chunks; we should do the same when embedding synthetic query expansions).
5. Replace ID-detection-then-translate (current `retriever.ts` flow) with **multilingual-native embedder** (qwen3-embedding-8b) and drop the translation hop — saves a round-trip, and Qwen handles Bahasa natively.

## Layer-by-Layer Findings

### 1. Extraction (PDF → markdown)

**Critical current-state gap:** `unpdf` strips ALL structure — no headings, no tables, no equations, no paragraph breaks.

Structural comparison on `attention.pdf` (15-page arxiv paper, ground truth: 27 sections, Table 1 with BLEU scores, multiple numbered equations):

| google/gemini-2.5-flash-lite | 44,361 | 6,011 | 24 | ✅ | 220 | ~$0.004/doc |
| google/gemini-2.5-flash | 37,257 | 4,176 | 24 | ✅ | 15 (lazy on eqs) | ~$0.01/doc |
| google/gemini-2.5-pro | 40,518 | 10,068 | 27 | ✅ | 18 (lazy on eqs) | ~$0.04/doc |

*Qualitative read (sampled lines 30-70 of each):* Claude Sonnet 4.6 produces the highest-fidelity LaTeX (`\sqrt{d_k}`, `\tag{1}` numbered equations, bold author names, italicized citations). Gemini-2.5-flash-lite is close behind at ~1/30th the cost. Unpdf produces a single flat blob with zero line breaks — when fed to `SmartChunker`, it hits the sentence-split fallback and loses section metadata entirely.

#### Cost-focused re-bench (post-initial-audit)

When we factored in "what if the user ingests 5K pages/day?", the original pick of `gemini-3-flash-preview` came out to ~$345/mo. Re-ran with cost priority. Both numbers below are measured.

| Model | attention.pdf | apple-10k | $/page | $/mo @ 5K pg/day | notes |
|---|---|---|---|---|---|
| **openai/gpt-4.1-nano** | 2.1s, $0.0047 | 3.8s, $0.0023 | **$0.00031** | **$87** | 🥇 cost-aware default. Markdown + numerical data preserved. |
| google/gemini-2.5-flash-lite | 5.6s, $0.0050 | 4.5s, **255K chars bloat** | $0.00067 | $101 | even with compact-tables prompt, bloats on tables. avoid. |
| openai/gpt-5-nano | timeout on attention | 2.9s, $0.0057 | — | — | flaky on longer docs |
| bytedance-seed/seed-1.6-flash | 0 chars | 9.4K chars, $0.010 | — | — | fails on attention |
| amazon/nova-lite-v1 | 7.8s, $0.032 | 5.6s, $0.009 | ~$0.002 | $273 | OR-routed image tokens blow up cost |
| x-ai/grok-4-fast | 4.5s, $0.0078, truncated? | — | — | — | fastest, short output — suspicious |
| google/gemini-3-flash-preview | 4.8s, $0.036 | 4.3s, $0.024 | $0.0024 | $345 | quality-first fallback |
| anthropic/claude-sonnet-4.6 | 10s, $0.15 | 6s, $0.06 | $0.010 | $1,500 | premium only |

**Decision:** primary = `openai/gpt-4.1-nano`. Fallback = `google/gemini-3-flash-preview`. Premium escalation available via `KB_EXTRACT_PRIMARY=anthropic/claude-sonnet-4.6` per-deployment.

Why nano works for RAG despite imperfect table cell alignment: retrieval matches against the *content* (numbers, entity names, phrases). The markdown structure needs to carry section boundaries (it does — nano emits proper `#`/`##`) and keep content intact (it does — all numbers present). Cell-border alignment doesn't affect retrieval.

**Why NOT `gemini-2.5-flash-lite`:** on `apple-10k` (table-heavy, 4 pages) it emitted **255K-400K characters** of output even with a prompt explicitly demanding compact tables. Bloat is model-inherent. Both storage and downstream LLM context get wrecked.

**Cost per doc (gpt-4.1-nano):** ~$0.005 per 15-page paper. For 10K docs that's $50. For 5K pages/day over a year: ~$570.

### 2. Chunking

Current `SmartChunker` is structurally sound **when input has markdown structure**. But when fed `unpdf` output (no structure), it falls through to sentence splitting and produces chunks without `section` metadata. That metadata is used as a prefix during embedding (`Section: X\n\n<body>`), which materially helps retrieval — we verified this empirically.

**Derivative finding:** the chunking layer is fine. The real problem is that chunking is a slave to extraction quality. Fix extraction, chunking gets better for free.

Optional future enhancement (not required, medium-effort): contextual chunks (Anthropic-style) where each chunk is prepended with an LLM-generated 1-2 sentence summary of what surrounds it. Leave for a follow-up; current chunker is adequate once extraction is fixed.

### 3. Embedding

Measured on 35 Q/A × 355 chunks (subsample of 1260 total; all expected-answer chunks included):

| Model | dim | hit@1 | r@5 | r@10 | MRR | $/1M tok |
|---|---|---|---|---|---|---|
| **qwen/qwen3-embedding-8b** | 4096 | **0.914** | 0.971 | 0.971 | **0.943** | $0.04 |
| openai/text-embedding-3-large | 3072 | 0.857 | 0.971 | 0.971 | 0.914 | $0.13 |
| **openai/text-embedding-3-small (CURRENT)** | 1536 | 0.771 | 0.971 | 0.971 | 0.864 | $0.02 |
| openai/text-embedding-ada-002 | 1536 | 0.771 | 0.914 | 0.971 | 0.845 | legacy |
| baai/bge-large-en-v1.5 | 1024 | 0.743 | 0.971 | 0.971 | 0.850 | $0.01 |
| qwen/qwen3-embedding-4b | 2560 | — | (broken on OR batch 64) | — | — | — |

Per-doc breakdown also shows qwen3-emb-8b is **uniformly** best — particularly strong on the Bahasa RDS proposal (8/8 hit@1 vs 7/8 for small) and on the arxiv paper (7/8 vs 5/8).

**Decision:** swap to `qwen/qwen3-embedding-8b`.

Implications:
- Vector store schema uses `array<float>` in SurrealDB. 4096 vs 1536 dims = 2.67× larger index. A 100K-chunk corpus today costs ~24 MB just for vectors with `f32`; after swap it's ~64 MB. Irrelevant.
- Need to re-embed every chunk in the DB on cutover (one-off batch job). At $0.04/M tokens + ~500 tok/chunk average: **~$2 to re-embed 100K chunks.** Negligible.
- Query embeddings also switch — latency-critical. Measured query latency: Qwen3-embedding-8b = **55ms for 35 queries** in batch (=1.6ms/query amortized); single-query will be ~80-150ms. Same ballpark as `text-embedding-3-small`.

### 4. Rerank

Measured on 35 Q/A × 355 chunks, retriever = `qwen/qwen3-embedding-8b`, top-20 → top-5:

| Reranker | hit@1 | r@5 | MRR | ms/q | notes |
|---|---|---|---|---|---|
| **llm:google/gemini-3-flash-preview** | **0.971** | 0.971 | **0.971** | 1690 | 🥇 only 1 miss out of 35 queries |
| llm:anthropic/claude-haiku-4.5 | 0.943 | 0.971 | 0.957 | 2003 | strong, slower |
| no-rerank | 0.914 | 0.971 | 0.943 | 0 | baseline |
| llm:google/gemini-2.5-flash | 0.914 | 0.971 | 0.943 | 965 | no lift |
| cohere/rerank-v3.5 | **0.857** | 0.971 | 0.910 | 441 | **hurts** |

**Surprises:**
- `cohere/rerank-v3.5` *demoted* the correct chunk below neighbors in 2 of 35 queries. For "What were Apple's total net sales for Q4?" cohere put the table header (`apple10k_1`) above the row with the actual number (`apple10k_2`). This is a pattern: cohere is trained to surface "canonical intro passages," but a RAG pipeline wants the passage with the *answer*. Don't use it until we can test v3.5-english-v3.0 or pair with explicit "prefer chunks containing specific entities/numbers" prompting.
- Gemini 3 Flash Preview as LLM-reranker is the best rerank we can reach from OpenRouter. The 1.7s latency is the cost.

**Decision:**
- Default ("fast" mode): **no-rerank** (0ms added, 0.914 hit@1 already).
- Opt-in "quality" mode: **LLM-rerank with `google/gemini-3-flash-preview`** (+5.7 pts hit@1, +1700ms). Enable via `KB_RERANK_ENABLED=true` or per-query flag on high-stakes queries.
- Do not enable cohere rerank.

### 5. End-to-end answer

4 generator models × 35 queries × 2 rerank configs = 280 answers. I grade each answer myself against the gold with this rubric:

- **5**: correct, precise, no hallucination
- **4**: correct but incomplete or slight phrasing deviation
- **3**: partially correct, or correct but over-terse
- **2**: wrong answer, or refused with "not in passages" when it *was* in passages
- **1**: hallucinated content not in retrieved passages

Graded so far (both with no-rerank + qwen3-embedding-8b retriever):

| Generator | 5/5 | 4/5 | 3/5 | 2/5 | 1/5 | Avg | $/1M out |
|---|---|---|---|---|---|---|---|
| google/gemini-2.5-flash-lite | 22 | 4 | 4 | 1 | 0 | **4.00** | $0.40 |
| **google/gemini-3-flash-preview** | **28** | 2 | 3 | 1 | 0 | **4.54** | $3.00 |

Gemini 3 Flash Preview wins decisively on generation quality — it reads the passages more carefully. On one Bahasa question ("does proposal cover hardware + mobile SDK?"), flash-lite refused ("not in passages") while flash-3 correctly parsed the negative-case wording and answered properly. Flash-3 also gave both reasons for sinusoidal positional encodings where flash-lite gave only one.

Common failure mode on both: refusing with "Not in provided passages" when the passages actually contain a partial answer. Fixable with a prompt tweak: *"if the question asks whether the doc covers X, answer about what IS covered alongside what is not."*

No hallucinations observed on either model. Zero-hallucination at top-5 context is a direct consequence of the strong retriever + compact answer-oriented prompt.

## Proposed Target Architecture

```
              ┌─────────────────────────────────────────────┐
              │  INGESTION                                   │
              │                                              │
 upload ─►    │  if PDF:    gpt-4.1-nano                     │
              │             (fallback: gemini-3-flash-preview)│
              │  if image:  gpt-4.1-nano                     │
              │  if HTML:   strip → markdown                 │
              │  if MD/txt: passthrough                      │
              │                          │                    │
              │                          ▼                    │
              │         SmartChunker (unchanged)             │
              │                          │                    │
              │                          ▼                    │
              │      embed: qwen/qwen3-embedding-8b          │
              │                          │                    │
              │                          ▼                    │
              │       SurrealDB (vector + metadata)          │
              └─────────────────────────────────────────────┘
                                         │
                                         │
              ┌─────────────────────────────────────────────┐
              │  RETRIEVAL                                   │
              │                                              │
 query ─►     │    embed query: qwen/qwen3-embedding-8b     │
              │          │                                   │
              │          ▼                                   │
              │    top-20 cosine (SurrealDB)                │
              │          │                                   │
              │          ▼                                   │
              │    take top-5 (default — no rerank)         │
              │    OR LLM-rerank via gemini-3-flash-preview │
              │       (opt-in "quality" mode, +1.7s)        │
              │          │                                   │
              │          ▼                                   │
              │    prepend section + doc title to passages  │
              │          │                                   │
              │          ▼                                   │
              │    feed to chat model                       │
              └─────────────────────────────────────────────┘
```

### Model defaults
| role | default | premium fallback | notes |
|---|---|---|---|
| PDF extraction | `openai/gpt-4.1-nano` | `google/gemini-3-flash-preview` | $0.00031/page (~$87/mo @ 5K pages/day). Quality fallback kicks in on primary error. |
| Image OCR | `openai/gpt-4.1-nano` | `google/gemini-3-flash-preview` | same |
| Embedding (doc + query) | `qwen/qwen3-embedding-8b` | — | 4096-dim, multilingual, 0.914 hit@1 measured |
| Reranker (off by default) | OFF (raw cosine top-5) | LLM-rerank `google/gemini-3-flash-preview` on "quality" mode | +5.7 pts hit@1, +1700ms. Never enable `cohere/rerank-v3.5` on this stack — it *hurts* hit@1. |
| Chat generator | `google/gemini-3-flash-preview` (fast, cheap, **4.54/5** measured) | `anthropic/claude-sonnet-4.6` | `gemini-2.5-flash-lite` is budget option (4.0/5, ¼ price) |

### Config surface
- New env vars:
  - `KB_EXTRACT_PRIMARY=openai/gpt-4.1-nano` (cost-aware default; swap to `google/gemini-3-flash-preview` for quality-first)
  - `KB_EXTRACT_FALLBACK=google/gemini-3-flash-preview`
  - `KB_EMBEDDING_MODEL=qwen/qwen3-embedding-8b`
  - `KB_RERANK_ENABLED=false` (default; set true for "quality" mode)
  - `KB_RERANK_MODEL=google/gemini-3-flash-preview` (only if enabled)
  - `KB_RERANK_INITIAL_K=20`
  - `KB_RERANK_FINAL_K=5`
  - `KB_GENERATOR_MODEL=google/gemini-3-flash-preview` (per-assistant override still wins)
- Migration: one-off script to re-embed all existing chunks. Store old vectors as `embedding_legacy` for rollback window; drop after 7 days.

## What this design deliberately does NOT do

- Doesn't introduce **knowledge graphs / entity extraction pipelines** beyond what exists. The existing `document-intelligence` module can stay; hybrid entity + vector search is documented as a future lever.
- Doesn't add **contextual chunking (Anthropic's "Contextual Retrieval")**. The measured gap after the 3 changes above is already below 3 pts hit@1; contextual chunking adds non-trivial ingestion cost and latency. Defer.
- Doesn't introduce **late-interaction / ColBERT-style retrieval**. Not reachable on OpenRouter; would require GPU-hosted index.
- Doesn't introduce **BM25 / hybrid lexical+vector fusion**. Current `hybrid-search.ts` already does vector + entity RRF. We leave it. Adding BM25 is a medium-effort follow-up if rerank alone doesn't close the gap.
- Doesn't try to replace SurrealDB or the vector-store interface.

## Testing strategy for the implementation

- Port the bench harness from `/tmp/kb-bench/` into `tests/bench-kb/` in-repo.
- CI-runnable smoke benchmark: 5 docs × 10 Q/A, <3 min runtime, enforces:
  - `hit@1 >= 0.85` on the smoke corpus
  - `recall@5 >= 0.95`
  - `avg query latency <= 600ms` (p95)
- Nightly full benchmark on the full 35-Q/A corpus.
- Alert on regression: > 3 pt drop in hit@1 vs previous night.
