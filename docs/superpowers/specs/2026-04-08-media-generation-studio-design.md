# Media Generation Studio — Design Spec

**Date:** 2026-04-08
**Status:** Draft for review
**Phase 1 scope:** Standalone Media Studio (image, audio, video) in the dashboard
**Phase 2 (deferred):** Expose generation as builtin LLM tools (`generate_image`, `generate_audio`, `generate_video`) and chat artifact renderers

## Overview

Add a Media Studio to the RantAI Agents dashboard for generating images, audio, and video through OpenRouter. The studio is a first-class workspace with prompt-driven generation, a model picker, an org-wide library of past generations, and reuse-as-input flows for image-to-image / image-to-video. Phase 2 will wrap the same backend service as builtin tools so assistants and digital employees can generate media inside chat. Phase 1 makes Phase 2 cheap by ensuring the service layer is callable from any context.

The studio uses OpenRouter as the sole provider, matching how chat models, embeddings, and the existing model sync are wired. Generated assets are stored in RustFS S3 (already in the docker compose stack). All generation is gated by per-user daily spending limits configurable in user settings.

## Goals

- Ship a usable image / audio / video generation studio behind one dashboard route
- Reuse existing infrastructure (OpenRouter, RustFS S3, Prisma, Socket.io, `LlmModel` sync, audit) — no new external services
- Build the backend service so Phase 2 (LLM tool wrappers) is a thin wrapping job, not a re-architecture
- Provide org-wide library with reuse-as-input from day 1
- Enforce per-user daily cost limits before any provider call

## Non-goals (Phase 1)

- LLM tool integration in chat (Phase 2)
- Custom (non-OpenRouter) provider adapters
- Org-wide budget caps (per-user limit only)
- Public sharing / embed of assets — assets stay org-internal
- Webhook callbacks from OpenRouter for video — we use cron polling
- Inline streaming of binary bytes — we always go through S3
- Automatic upscaling, background removal, face restore, or other post-processing tools
- Per-user-private generations (org-scoped library only; private toggle is one boolean field away)

## Provider & SOTA notes (April 2026)

| Modality | Status | Notable models |
|---|---|---|
| Image | GA | Nano Banana 2 (Gemini 3.1 Flash Image), Nano Banana Pro, GPT-5 Image, GPT-5 Image Mini, Seedream 4.5, Riverflow V2 Pro |

## Data model

New Prisma models added to `prisma/schema.prisma`:

```prisma
enum MediaModality {
  IMAGE
  AUDIO
  VIDEO
}

enum MediaJobStatus {
  PENDING       // queued, before provider call
  RUNNING       // provider call in flight or polling
  SUCCEEDED
  FAILED
  CANCELLED
}

model MediaJob {
  id                 String          @id @default(cuid())
  organizationId     String
  userId             String          // who triggered it
  modality           MediaModality
  modelId            String          // OpenRouter model id
  prompt             String          @db.Text
  parameters         Json            // model-specific (size, voice, duration, etc.)
  referenceAssetIds  String[]        @default([])  // image-to-* / edits
  status             MediaJobStatus  @default(PENDING)
  providerJobId      String?         // openrouter video poll id
  errorMessage       String?         @db.Text
  estimatedCostCents Int?            // computed at gate time
  costCents          Int?            // populated on completion
  startedAt          DateTime?
  completedAt        DateTime?
  createdAt          DateTime        @default(now())
  updatedAt          DateTime        @updatedAt

  organization       Organization    @relation(fields: [organizationId], references: [id], onDelete: Cascade)
  user               User            @relation(fields: [userId], references: [id])
  assets             MediaAsset[]

  @@index([organizationId, createdAt])
  @@index([userId, createdAt])
  @@index([status])
  @@index([modality])
}

model MediaAsset {
  id                 String         @id @default(cuid())
  jobId              String
  organizationId     String
  modality           MediaModality
  mimeType           String         // image/png, audio/mpeg, video/mp4
  s3Key              String         // RustFS path
  sizeBytes          Int
  width              Int?
  height             Int?
  durationMs         Int?           // audio/video
  thumbnailS3Key     String?        // generated for video
  isFavorite         Boolean        @default(false)
  metadata           Json?          // model id, seed, raw provider response excerpt
  createdAt          DateTime       @default(now())

  job                MediaJob       @relation(fields: [jobId], references: [id], onDelete: Cascade)
  organization       Organization   @relation(fields: [organizationId], references: [id], onDelete: Cascade)

  @@index([organizationId, createdAt])
  @@index([jobId])
  @@index([modality])
}
```

Extension to existing `User` model:

```prisma
model User {
  // ... existing fields
  mediaLimitCentsPerDay Int?       // null = no limit
}
```

Extension to existing `LlmModel` model (added in commit `348b578` earlier this session):

```prisma
model LlmModel {
  // ... existing fields
  outputModalities String[]         @default([])  // ["text"], ["image"], ["audio"], ["video"]
  inputModalities  String[]         @default([])  // ["text", "image"], etc.
}
```

The existing `cron/sync-models` job already fetches the full OpenRouter model list; it just needs a small extension in `src/lib/models/sync.ts` to read the `architecture.input_modalities` and `architecture.output_modalities` fields and write them into the new columns. No new cron job, no new fetch, no new auth — just two extra field reads per model.

### Why split MediaJob and MediaAsset

One job can produce multiple assets — many image models return 1-4 variants from a single prompt. Assets must be addressable independently for the library and for the reuse-as-input flow. Splitting also keeps the job row small (no binary metadata) and lets cascading deletes work cleanly when an asset is removed.

### Why `referenceAssetIds` is an array, not a join table

Cardinality is small (1-4 references per job). A join table would add a query for every list page and a write for every job. The array preserves order (which matters for some models — primary reference vs secondary) and is queryable via Prisma's array operators when needed.

### Cron route

```
src/app/api/cron/media/poll-video-jobs/route.ts
src/features/platform-routes/cron-poll-video-jobs/service.ts
```

Runs every 30 seconds. Finds `MediaJob` rows where `modality = VIDEO` and `status = RUNNING`, polls OpenRouter using `providerJobId`, advances to `SUCCEEDED` (with asset upload) or `FAILED`. Survives restarts. Matches the existing cron pattern (`cron/workflows`, `cron/sync-models`).

### Optimistic UI rule

The studio renders the asset the moment `POST /jobs` returns it (sync image/audio path). Socket.io updates are still emitted — they become a no-op for the originating client because the local store already has the asset, but other tabs and other users in the same org still receive the update in real time.

For the async video path, the studio shows a "Generating…" placeholder card with elapsed time and an estimated remaining duration. The placeholder is replaced when the Socket.io event arrives.

### Per-user limit enforcement

Single function `enforceMediaLimit({ userId, estimatedCostCents })` called at the top of `POST /jobs`:

1. Read `user.mediaLimitCentsPerDay`. If `null`, allow.
2. Sum `MediaJob.costCents` for `userId` where `createdAt > now - 24h` and `status` in `(SUCCEEDED, RUNNING)`. For `RUNNING` rows, prefer `estimatedCostCents`.
3. Add `estimatedCostCents` (computed from `LlmModel.pricingOutput` × parameters — for video this is `pricePerSecond × duration`).
4. If sum exceeds limit, return `402 Payment Required` with `{ error, limit, used, requested }`.

Cost is **estimated up front** for the gate and **finalized on completion** from OpenRouter's response. The limit is enforced on estimate so a runaway video generation cannot bypass the cap.

## Phase 2 hook: tool integration contract

Phase 2 adds three thin builtin tool wrappers around the same service:

```
src/lib/tools/builtin/
├── generate-image.ts     ← new in Phase 2
├── generate-audio.ts     ← new in Phase 2
└── generate-video.ts     ← new in Phase 2
```

Each is approximately 40 lines. Inside, they call the same `createMediaJob` function the studio uses. The only difference is the caller context — `userId` becomes the assistant's resolved user, `organizationId` is the assistant's org, the cost flows through the same per-user limit.

### Artifact renderers

The existing artifact system (`src/features/conversations/components/chat/artifacts/`) currently supports:

```
text/html, text/markdown, image/svg+xml, application/react,
application/mermaid, application/code, application/sheet,
text/latex, application/slides, application/python, application/3d
```

Phase 2 adds three new artifact types:

```
image/png      (or image/jpeg, image/webp)
audio/mpeg     (or audio/wav)
video/mp4
```

Plus three new renderers in `chat/artifacts/renderers/`:

- `image-asset-renderer.tsx` — wraps `<img>` with download / fullscreen / "open in studio"
- `audio-asset-renderer.tsx` — `<audio>` with waveform visualization
- `video-asset-renderer.tsx` — `<video>` with poster from `MediaAsset.thumbnailS3Key`

These renderers store an `assetId`, never the bytes. The renderer fetches metadata from `/api/dashboard/media/assets/[id]` and renders from a presigned download URL. This keeps chat history small, makes deletes consistent, and lets the same asset appear in many chats and the studio library simultaneously.

### Phase 2 contract — invariants Phase 1 must satisfy

| Invariant | Why |
|---|---|
| `createMediaJob` is callable from any context (HTTP, server action, tool) — no Next.js coupling | So the tool can call it from `lib/tools/builtin/` without circular imports |
| Per-user limit enforcement lives in the service, not in the route handler | So tools cannot accidentally bypass the limit |
| `MediaAsset` is identified by stable cuid, never by URL | So chat artifacts can reference assets that may move S3 paths |
| Costs are written to the existing audit table from the service | So tools and studio share one audit trail |
| Job creation emits a Socket.io event regardless of caller | So future tool calls can stream "image generating…" status into chat |

These invariants will be encoded as integration tests in the Phase 1 plan so they cannot drift.
